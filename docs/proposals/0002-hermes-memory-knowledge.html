Why 0002 (the strategic argument)

Hermes 而家係 stateless planner。每個新 project:

  1. Planner LLM 由 zero 發明 7-step plan
  2. Agent 重複踩同一啲坑(DB race / flaky test / wrong import path)
  3. 失敗嘅 root cause 留喺 journal.log,下個 project 完全睇唔到
  4. Cost 預測 = 估個數 + budget buffer
  5. Agent prompts 每個 project 重新寫 boilerplate

10 個 project 之後,你已經累積咗 useful knowledge,但 Hermes 唔知道。 本提案將呢啲 scattered wisdom 變成 queryable, retrievable, applied 嘅 institutional memory。

10x 效果: 第 N 個 project 嘅成功率 / cost-efficiency 應該 asymptotically approach 100% / optimal,
因為每次失敗都會 refine 下次嘅 bootstrap。

Pain point

0001 解決咗「睇唔到」嘅問題,但仲有:

  • ❌ 冇 institutional memory — planner LLM 每個 project 重新發明輪子
  • ❌ 失敗 patterns 唔累積 — 同一個 bug category(DB race / flaky test / type error)重複撞
  • ❌ Cost 預測靠估 — "呢個 task 大概幾錢"冇 data 答你
  • ❌ Cross-project search 唔可能 — "之前做過類似嘅嘢喺邊"搵唔到
  • ❌ Agent prompts 重複 engineer — boilerplate 每個 project 重寫
  • ❌ 冇 postmortem — 失敗 project 嘅 root cause 冇 structure 紀錄

Implementation

  • Storage: bun:sqlite + FTS5 virtual table (built-in, zero deps)
  • Schema: memories(id, type, content_md, tags JSON, source_project, use_count, success_count, created_at, last_used_at)
  • Index: FTS5 on content_md + tags; B-tree on type, last_used_at
  • MVP retrieval: FTS5 keyword match + tag boost + recency boost + use_count boost (hand-tuned scoring)
  • Phase 2 retrieval: add embeddings (voyage-3 or text-embedding-3-small) stored as BLOB; cosine sim in JS (no vector DB needed for <10k entries)
  • Auto-extract triggers:
    1. Project status → completed: planner reviews journal, proposes 1-3 patterns + 0-2 snippets
    2. Project status → failed: failure analyzer generates 1+ lesson per failure category
    3. Human review in APP (thumbs up/down) before persisted to vault

B · Smart Bootstrap

Flow

User: /project start "add OAuth login with Google"

Planner LLM (with memory context):
  ↓
1. Bootstrap query: classify goal → "OAuth + Google"
2. Retrieve top-K=5 memories (keyword + tag + recency)
   → 3 patterns (OAuth setup ×2, env var mgmt ×1)
   → 1 snippet (Google OAuth client init)
   → 1 lesson ("Don't store refresh tokens in JWT")
  ↓
3. Inject into planner prompt as <memories> block
  ↓
4. Planner generates plan, citing memories by id
  ↓
5. Saved plan.md includes "📚 Referenced memories" section
  ↓
6. User reviews in APP (0001 P2.5 approval flow)
   → can see, edit, replace memories used

Implementation details

  • Retrieval: hybrid score = 0.6 * fts_score + 0.25 * tag_match + 0.10 * recency + 0.05 * use_count
  • Token budget: top-5 memories, max 2000 tokens total injected — measured, not guessed
  • Cold start: with <10 memories, retrieval still runs (returns empty gracefully)
  • User override: 0001 P2.5 plan review shows referenced memories; user can replace / remove / add
  • Citation: planner MUST cite memory ids in output → enables "show source" link in plan

Implementation

  • Event sourcing: every project writes structured events to events.jsonl(task start/end, agent used, cost, retry count)
  • Aggregation: nightly job (or on-demand via API) computes stats by dim(task_type, agent, model)
  • Postmortem trigger: on status → failed, run haiku summarizer over last 20 journal entries + error logs
  • UI: GET /api/stats + dashboard page in 0001 APP (reuses HTMX scaffold)

D · Cost Predictor

Predict API

POST /api/predict
  { "plan": [...tasks], "goal": "add OAuth login" }
  ↓
{
  "predicted_cost_usd": 1.20,
  "predicted_cost_stddev": 0.40,
  "predicted_wall_min": 28,
  "predicted_wall_stddev": 8,
  "success_probability": 0.85,
  "based_on": 12,            // similar past projects
  "novelty_score": 0.15,     // 0=seen-it-all, 1=brand-new
  "warnings": [
    "Task type 'frontend-UI' has 71% historical success rate",
    "Plan includes 2 retries on prior similar runs"
  ]
}

Method

  1. Embed new plan via task-type tags + free-text goal
  2. Find K=12 most similar past projects (by tag overlap + cos sim if embeddings)
  3. Weighted average cost / wall / success rate (weight by similarity)
  4. Bootstrap confidence interval (n=12 small, so widen by 1.5×)
  5. Novelty score: 1 - (max similarity to any past project)
  6. Warnings: surface any historical pitfall on matched task types

Data model additions

New SQLite tables (separate file vault.db)

CREATE TABLE memories (
  id TEXT PRIMARY KEY,                 -- mem-{nanoid(8)}
  type TEXT NOT NULL,                  -- 'pattern' | 'snippet' | 'lesson'
  title TEXT NOT NULL,
  content_md TEXT NOT NULL,
  tags TEXT NOT NULL,                  -- JSON array
  source_project TEXT,                 -- project_id, nullable for manual
  use_count INTEGER DEFAULT 0,
  success_count INTEGER DEFAULT 0,
  failure_count INTEGER DEFAULT 0,
  embedding BLOB,                      -- Float32Array, populated M5
  created_at TEXT NOT NULL,
  last_used_at TEXT
);

CREATE VIRTUAL TABLE memories_fts
  USING fts5(title, content_md, tags,
             content='memories', content_rowid='rowid');

CREATE TABLE memory_uses (             -- for analytics
  memory_id TEXT,
  project_id TEXT,
  task_id TEXT,
  outcome TEXT,                        -- 'success' | 'fail' | 'pending'
  used_at TEXT,
  PRIMARY KEY (memory_id, project_id, task_id)
);

CREATE INDEX idx_memories_type ON memories(type);
CREATE INDEX idx_memories_last_used ON memories(last_used_at DESC);

Out of scope (defer)

  • Multi-user memory isolation — single user assumption (0001 嘅 choice)
  • Memory versioning — humans edit in place; no history
  • Auto-pruning / decay — manual only; trust the user
  • Cross-vault federation — no sharing between Hermes instances
  • External knowledge import — no scraping docs / StackOverflow
  • Memory as fine-tune data — just retrieval, not training
  • Real-time collaborative editing — single user
