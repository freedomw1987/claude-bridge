## What was built

`/project setMode auto <duration>` — user-bounded autonomous windows for
Hermes. The user types e.g. `setMode auto 30m` and the bot arms a
`setTimeout` that calls `softExit(reason="duration_expired")` at the next
judge boundary when the deadline passes.

| Sub-system | Files | What it does |
|---|---|---|
| `duration.ts` | 1 | `parseDuration` ("30m", "1h30m", "1d") + `formatCountdown` + 365d cap |
| `types.ts` | 1 | `ProjectTimer`, `KilledReason`, `JournalEntryType "timer"` |
| `state.ts` | 1 | `stripTimerHandle` / `clearTimer` (handle is transient, never persisted) |
| `orchestrator.ts` | 1 | `checkTimerExpired` (judge boundary) + `softExit` (idempotent) + `armProjectTimer` (with past-deadline microtask fallback) |
| `hermesCommands.ts` | 1 | `matchSetMode` regex with duration + 4 branches in `handleProjectSetMode` (auto/manual + active/terminal) |
| `discord.ts` | 1 | `formatStatusEmbed` ⏱ Timer line + `formatTimerExpired` post |
| Tests | 4 | duration (42) + types (8) + state (11) + orchestrator (15+3 E2E) + discord (7) + hermesCommands (10) = **82 new** |
| Docs | 4 | ADR-0004, PRD F7, ARCHITECTURE state-machine diagram, MILESTONES M2.1–M2.11, REGRESSION-GUARD RG-001, this retro |
| Script | 1 | `scripts/smoke-m2.sh` — manual Discord walkthrough (now fallback only) |

**Net**: ~2,000 LOC added, 0 lines deleted in production code, 0 typecheck errors, 0 regressions across the existing 222-test suite.

## What worked

### 1. Plan + ADR lock before any code (`f04630f`)

Committed the 11-task plan + ADR-0004 to git *before* writing a single
line of implementation. The ADR-0004 "Why soft-exit at judge verdict,
not at task boundary" section (idempotency / cost / alignment with
existing judge intent) was the single most useful piece of design
writing in the whole phase — every subsequent implementation choice
pointed back at it.

**Carry forward**: Any feature that has more than 1 axis of trade-off
(soft vs hard exit, polling vs event, sync vs async) should ADR-lock
the trade-off first.

### 2. Surgical 50–100 line commits per sub-task

Each M2.x landed in 1–2 files with a self-contained commit message that
named the sub-task ID. When something broke (LSP cache on type
assertions, `JSON.stringify(Timeout) → {}` bug, `softExit` ENOENT on
never-resumed projects), the bisect was a 30-second `git log --oneline
--grep M2.x`.

**Carry forward**: The "one M-x per commit" pattern is the right
granularity. M2.6 (the user-facing surface) ate M2.7 (active-state
gate relaxation) as a side-effect, but the commit message explicitly
called this out — *that's* the rule, not "one commit per micro-task".

### 3. Edge cases caught by tests, not by users

Three silent-failure modes were caught by the test suite before they
shipped:

| Edge case | Caught by | Impact |
|---|---|---|
| `JSON.stringify(Timeout) → {}` writes `{handle: {}}` to disk | `types.test.ts` M2.2 | Would have crashed `loadState` on the next bot restart |
| `softExit` on never-resumed project → `state.json` save fails ENOENT | `orchestrator.test.ts` M2.4 | Would have silently swallowed the kill |
| `parseDuration("365d1h")` returns `null` but "365d" passes | `duration.test.ts` M2.1 | Would have given the user a confusing "Capped at..." instead of "Invalid duration" |

**Carry forward**: Always write the "bug" test FIRST in red, even if
you just learned about the bug 30 seconds ago. The 5 minutes saved by
"just adding the test for the working case" is never worth the 2 hours
of prod-debugging later.

### 4. Mock-free integration via real `state.json` round-trip

The M2.11 E2E test creates a real `mkdtempSync` directory, calls
`saveState`, `armProjectTimer`, waits 400ms, then reads back from disk
and asserts. No mocks of `Date.now`, no mocks of the file system, no
mocks of `setTimeout`. Total test runtime: ~470ms for 3 tests.

**Carry forward**: When you find yourself reaching for `vi.useFakeTimers()`
or `jest.mock("fs")`, ask first: "is the real behavior slow, or is it
*correct*?" Real behavior was 400ms — fast enough. A mock would have
hidden the exact thing the test was supposed to catch (the disk-level
state transition).

### 5. Regression Guard entry (RG-001) locked 8 invariants

The RG-001 entry doesn't just say "we fixed a bug" — it lists 8
invariants (I-1..I-8) with the test that pins each one. Refactor
procedure documented in the same file. This is the difference between
"we tested this once" and "this is permanent".

**Carry forward**: Every feature should ship a regression-guard entry
that names invariants, not just tests. Tests can be deleted; invariants
are what humans (and AI subagents 6 months from now) need to preserve.

## What hurt

### 1. `JSON.stringify(Timeout)` bug discovered AFTER shipping the type

`ProjectTimer` was added in M2.2 with a `handle: Timeout` field, then
discovered that `JSON.stringify(new Timeout())` returns `"{}"` (not
`undefined`). The fix in M2.3 added `stripTimerHandle` to the save
pipeline, but if anyone had done a manual `cat state.json` between
M2.2 and M2.3, they would have seen `{handle: {}}` sitting in the
file and panicked.

**Lesson**: When adding a field that contains a non-serializable
reference (Timer, Socket, Stream), add the `JSON.stringify` round-trip
test in the SAME commit. The types-only commit is misleading — it's
not done until persistence is also done.

### 2. LSP cache lied about typecheck status 4+ times

Across M2.5/M2.6/M2.11, the LSP server reported stale errors that
disappeared after `tsc --noEmit` ran in the terminal. Wasted maybe 20
minutes total on "fixing" issues that didn't exist.

**Lesson**: The bash `bun run typecheck` is the source of truth. LSP
diagnostics are advisory only. When in doubt, run typecheck in the
terminal — the LSP lag is real and the typecheck is sub-200ms.

### 3. M2.6 file path correction in commit message

The M2.6 commit message said "in src/hermes/hermesCommands.ts" but
the actual file path is `src/discord/handlers/hermesCommands.ts`. The
test file (`hermesCommands.test.ts`) was in the right place, so the
tests passed and shipped, but a future grep for the wrong path would
miss the change.

**Lesson**: When patching a file that lives in a different directory
than the plan doc says, double-check with `find` before writing the
commit message. The plan doc is from session-day-1; reality is
session-day-N. Reality wins.

### 4. `await new Promise(r => setTimeout(r, 400))` felt like cheating

The M2.11 E2E test literally waits 400ms in a unit test. The reflex
was to use `vi.useFakeTimers()` to fast-forward. But fake timers would
have tested "we can schedule a callback that calls `softExit`", not
"the *real* `setTimeout` + *real* `softExit` + *real* disk write chain
produces a `killed` state at the deadline". The 400ms is part of the
test's *correctness* claim, not its slowness.

**Lesson**: Real wallclock time is sometimes the right tool. The
"fast unit test" reflex is a cargo cult when the integration you're
testing depends on real timing.

### 5. M2.11 started as "manual Discord smoke" — got replaced mid-phase

The original M2.11 plan was a 70-second manual walkthrough in Discord.
We shipped `scripts/smoke-m2.sh` with that assumption. Then David
asked for M2.11 to ship, which forced the question: "do we ship the
manual walkthrough, or do we automate it?" Automation won because it
runs in CI, runs in <500ms, and doesn't require a Discord bot. The
manual script is now a fallback.

**Lesson**: When a task is "manual smoke", ask "can I script this in
<1 second with the tools I already have?" before committing to a
walkthrough doc. If yes, the script is better and the walkthrough is
the fallback.

## Decisions revisited

| Decision | Verdict | Notes |
|---|---|---|
| Soft-exit at **judge verdict**, not at task boundary | ✅ Right | Idempotent + aligns with existing intent check. Long tasks still run to completion, but that's a known + acceptable trade-off (10min task finishing 2min after timer = fine, vs cutting it off mid-LLM-call = bad) |
| Default to `HERMES_MAX_WALL_HOURS` (4h) when user types `setMode auto` with no duration | ✅ Right | Matches the existing safety cap; user who wants something different types the duration |
| Clamp `effective = min(user, cap)` and surface "Capped at X" | ✅ Right | Surfaces the cap rather than silently truncating; user can `setMode auto 30m` repeatedly if they want multiple windows |
| State field `timer.handle` is **transient** (stripped on save, hydrated to null on load) | ✅ Right | `setTimeout` handles can't be JSON-serialized. The strip-and-rehydrate pattern is the cleanest way to handle this; alternative was "don't store expiresAt and recompute on boot" which loses 1 sub-second precision per restart |
| `softExit` is **idempotent** (judge boundary + setTimeout can both fire) | ✅ Right | Both paths do the same mutation, and `isActive` gate in the setTimeout callback prevents double-save |
| Resume-on-startup **does not** auto-restore the old `setTimeout` handle from disk | ✅ Right | A persisted handle would be a `{}` object after `JSON.stringify` round-trip; `armProjectTimer` schedules a fresh `setTimeout(expiresAt - now)` from the persisted wallclock ms |
| Single 4-line regex in `matchSetMode` with optional duration group | ⚠️ Re-examine | Works for "30m", "1h30m", "1d" but rejects "1d12h" because `[dhms]+` is greedy and the parser's `MAX_DURATION_MS` of 365d means a 376d request returns null. A real test of "1d12h" passes parseDuration, but the regex captures it as `1d` and leaves `12h` as trailing garbage. **Bug**: handleProjectSetMode treats "1d12h" as malformed. Need to either (a) make the regex `[dhms]+` or (b) call parseDuration on the captured `m[2]` and re-validate. Filed as post-M2-1 polish. |

The last item is a real, low-severity bug. It doesn't break anything in
the current spec (4h cap < 1d), but it's a sharp edge. Adding to the
post-M2 polish list.

## Lessons (concrete rules for next phase)

1. **Plan + ADR lock before code** for any feature with >1 trade-off axis. The ADR's "Why we picked X over Y" section is the most valuable artifact of the whole feature.
2. **One sub-task per commit**, but sub-tasks can be merged if a downstream sub-task is a trivial consequence of an upstream one. Always call out the merge in the commit message.
3. **Round-trip tests in the same commit as the type.** Adding a non-serializable field (`Timeout`, `Socket`, etc.) requires the save/load round-trip test to be in the same commit as the type addition.
4. **Real wallclock > fake timers** when the integration depends on real timing. The 400ms is correctness, not slowness.
5. **`bun run typecheck` is the source of truth**, not LSP diagnostics. When in doubt, re-run typecheck.
6. **File paths in commit messages**: verify with `find` before committing, especially when the plan doc was written on a different day.
7. **Regression-guard entries list invariants, not tests.** Tests can be deleted; invariants must be preserved through refactor.
8. **"Can I script this in <1 second?"** — ask before writing a manual walkthrough. The walkthrough is the fallback, not the primary deliverable.
